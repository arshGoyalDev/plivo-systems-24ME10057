Flaky Network Streaming System Summary

Created by your coding agent. This document summarizes the architecture, design choices, and system-level behavior of the sender/receiver implementation in systems_handout.

Executive Summary

The assignment is a real-time UDP media transport problem. The harness produces one 160-byte frame every 20 ms, sends it to the student sender, passes all sender/receiver traffic through a hostile relay, and grades whether the receiver delivers the correct frame to the player before t0 + DELAY_MS + seq * 20ms. The relay can drop, delay, duplicate, and reorder packets, and every byte sent through the relay counts against a strict 2.0x overhead cap.

The implemented design uses a hybrid reliability strategy: small-group XOR forward error correction handles common single-packet losses without waiting for a round trip, while deadline-aware NACK retransmission handles gaps, burst losses, and lost parity. The receiver runs a timed event loop so recovery decisions are tied to playout deadlines rather than simple packet arrival order.

The harness legs use the fixed assignment format of a 4-byte big-endian sequence number followed by a 160-byte payload. The internal protocol between the sender and receiver is custom and consists of three packet types:

Packet Direction Wire Format Purpose
TYPE_DATA = 0x01 Sender to receiver 1-byte type, 4-byte sequence, 160-byte payload Carries each original media frame across the relay.
TYPE_PARITY = 0x02 Sender to receiver 1-byte type, 4-byte group base, 1-byte group size, 160-byte XOR payload Lets the receiver reconstruct exactly one missing frame from a two-frame group.
TYPE_NACK = 0x10 Receiver to sender 1-byte type, 4-byte requested sequence Requests retransmission when FEC alone cannot recover a missing frame before deadline.

Sender Design

sender.c binds to the harness source on port 47010 and to relay feedback on port 47004. It forwards each source frame to the relay as a TYPE_DATA packet and stores the payload in a rolling retransmission buffer of 128 slots. The slot is selected with seq & (RETX_BUF - 1), and the stored sequence number is checked before retransmission so stale slots are not used for a different frame.

The sender also maintains a two-frame XOR accumulator. For every two consecutive data frames, it emits one TYPE_PARITY packet containing the XOR of the two payloads. The group size of two is a deliberate latency choice: parity becomes available after only one extra 20 ms frame interval, which keeps FEC useful under the target playout delay.

Feedback handling is intentionally simple and aggressive. When a valid TYPE_NACK arrives and the requested frame is still in the retransmission buffer, the sender immediately resends the original data packet. There is no one-retransmission-only guard, because the relay can also drop NACKs or retransmissions.

Receiver Design

receiver.c is the core of the reliability logic. It binds to relay media on port 47002, sends recovered frames to the harness player on port 47020, and sends NACK feedback through the relay on port 47003. Its main loop uses select() with short sleeps so the process can receive packets and also perform deadline checks without blocking at the wrong moment.

Incoming data frames are stored in a 2048-slot rolling frame window. The receiver separately tracks whether a frame is present and whether it has already been forwarded, which handles duplicate relay deliveries without sending the same frame to the player multiple times. Incoming parity packets are stored by group and immediately used to attempt recovery.

FEC recovery works by starting with the parity XOR payload and XORing every present frame in the group. If exactly one frame is missing, the remaining bytes are the missing payload. The receiver stores the reconstructed frame and forwards it if it has not already been delivered.

ARQ is deadline-aware. The receiver sends a NACK when it detects a gap, when parity reveals a missing group member, or when the next expected frame is approaching its deadline. Repeated NACKs are allowed with a 40 ms cooldown, but only while enough time remains for the feedback and retransmission to cross the relay. This trades a small amount of feedback bandwidth for much better recovery under moderate loss.

Timing Model

The receiver reconstructs the same playout schedule used by the harness: t0 + delay_s + seq * 0.020. It uses a small output margin so frames are sent slightly before their exact deadline. A next_check pointer advances through frame sequence numbers and performs last-chance FEC and forwarding checks as deadlines pass.

This timing model matters because the relay can reorder and delay packets. The receiver cannot assume that a later sequence means an earlier sequence is permanently lost. Instead, it keeps a jitter/recovery window and uses the absolute deadline to decide when waiting is no longer useful.

Design Choices and Tradeoffs

  • XOR FEC instead of larger erasure coding: XOR over two frames is cheap, standard-library-only, and fast enough for a 20 ms frame cadence. It costs roughly one extra parity packet per two data packets.
  • Small FEC group size: GROUP_SIZE=2 minimizes parity generation delay. Larger groups would reduce overhead but would delay parity and make recovery less useful before strict playout deadlines.
  • NACK-based ARQ as a second layer: FEC cannot recover when two packets in the same group are lost or when parity is lost. NACKs provide targeted repair without blindly duplicating every frame.
  • Repeated NACKs with cooldown: The feedback path is also lossy, so one NACK is not reliable. The 40 ms cooldown bounds feedback overhead and avoids flooding the relay.
  • Rolling buffers: Fixed-size arrays avoid dynamic allocation in the hot path and keep memory ownership clear in C. Sequence numbers are stored with slots to defend against wraparound collisions.
  • Immediate forwarding when possible: When a correct data or recovered frame is available, it is sent to the player rather than artificially waiting. The deadline loop is used for recovery checks, not for adding avoidable latency.

Observed Results

The development log records a progression from duplicate suppression, to XOR FEC, to deadline-aware playout checks, to NACK retransmission, and finally to aggressive ARQ. A verification run on Profile B at delay_ms=100 was valid, with 9 misses out of 1500 frames, a 0.60% miss rate, and 1.86x overhead. The saved relay artifact also shows 443,610 upstream bytes and 2,420 feedback bytes, which is about 1.86x the raw 240,000-byte stream.

Recommended grading setting: delay_ms = 100, as stated in NOTES.md. The design is tuned to remain under the 1% deadline-miss cap and 2.0x bandwidth cap while handling the known mild and moderate profiles.

System-Level Failure Modes

The architecture is optimized for bounded jitter and moderate loss, not arbitrary outages. It can fail if burst losses remove both data frames and parity from a group, if both NACKs and retransmissions are repeatedly dropped, or if relay delays are consistently high enough that retransmitted data cannot return before the playout deadline.

The retransmission buffer holds 128 frames, about 2.56 seconds of media at 20 ms per frame. Losses detected after that window cannot be repaired by ARQ because the sender may have overwritten the old payload slot. The chosen 100 ms delay is therefore a practical compromise: large enough for parity and some retransmissions, but still low enough to compete on playout delay once validity is achieved.
